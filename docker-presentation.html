Module 1

Microservices & Cloud Native

Monolith vs Microservices

Monolithic Architecture

  • Single, unified application
  • All components tightly coupled
  • One codebase, one deployment
  • Scales as a single unit

Microservices Architecture

  • Independent services
  • Loosely coupled components
  • Deploy services independently
  • Scale individual services

Key Insight: Each microservice handles a specific function and can have its own database

Why Stateless Applications?

The cloud favors stateless applications for predictable and easy scaling.

Benefits of Stateless Design

  • Horizontal Scaling: Traffic can be routed to any available instance
  • Elasticity: Handle swings in demand easily
  • Multi-Region: No need to sync session data
  • Resilience: Failed instances can be replaced without data loss

Principle: State and logic are decoupled - store state externally (databases, caches)

What is Cloud Native?

Architecture

  • Microservices
  • API-first communication
  • Externalized state

Packaging

  • Containerization
  • Immutable images
  • Version controlled

Operations

  • Automated orchestration
  • Infrastructure as Code
  • CI/CD pipelines

Infrastructure as Code (IaC)

  • Reproducible, declarative, text-based configurations
  • Version controlled with code reviews and PRs
  • Automated provisioning and configuration

Module 2

Introduction to Containers

Containers vs Virtual Machines

Virtual Machines

  • Full OS per instance
  • Gigabytes in size
  • Minutes to start
  • Hardware-level isolation
  • Requires hypervisor

Containers

  • Shared OS kernel
  • Megabytes in size
  • Seconds to start
  • Process-level isolation
  • Container runtime (Docker)

Key Benefit: "It works on my machine" becomes "It works in this container" - consistent behavior everywhere

Container Ecosystem

Runtimes

  • Docker
  • containerd
  • CRI-O
  • Podman

Orchestration

  • Kubernetes
  • Docker Swarm
  • Amazon ECS

Tools

  • Helm
  • Prometheus
  • Grafana

Docker Architecture

Key Concepts

  • Image: Read-only template for creating containers
  • Container: Running instance of an image
  • Dockerfile: Script defining how to build an image
  • Registry: Storage for Docker images (Docker Hub, ECR)
  • Volume: Persistent storage for container data

Module 3

Building Docker Images

Writing a Dockerfile

# Use an official base image
FROM python:3.9-slim

# Set working directory
WORKDIR /app

# Copy requirements first (for caching)
COPY requirements.txt .

# Install dependencies
RUN pip install -r requirements.txt

# Copy application code
COPY . .

# Expose port
EXPOSE 5000

# Define startup command
CMD ["python", "app.py"]

Understanding Layers

Each Dockerfile instruction creates a new layer. Layers are cached for efficiency.

Layer Best Practices

  • Order instructions from least to most frequently changing
  • Combine RUN commands to reduce layers
  • Copy dependency files before source code
  • Use .dockerignore to exclude files
# Good: Combined RUN
RUN apt-get update && \
    apt-get install -y nginx && \
    rm -rf /var/lib/apt/lists/*

# Bad: Multiple layers
RUN apt-get update
RUN apt-get install -y nginx
RUN rm -rf /var/lib/apt/lists/*

Multi-Stage Builds

Create smaller production images by separating build and runtime stages.

# Build stage
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o myapp

# Production stage - minimal image
FROM alpine:latest
COPY --from=builder /app/myapp /myapp
CMD ["/myapp"]

Result: Final image contains only the compiled binary, not the entire Go toolchain

Creating Efficient Images

Base Image Comparison

Base Image Size Use Case
ubuntu:22.04 ~77MB Full-featured development
debian:slim ~52MB Reduced footprint
alpine:latest ~7MB Minimal production images
distroless ~2MB Maximum security

Tip: Use specific version tags, not :latest, for reproducible builds

Essential Docker Commands

# Build an image
docker build -t myapp:v1 .

# List images
docker images

# Run a container
docker run -d -p 8080:80 --name myapp myapp:v1

# List running containers
docker ps

# View logs
docker logs myapp

# Execute command in container
docker exec -it myapp /bin/sh

# Stop and remove
docker stop myapp && docker rm myapp

Module 4

Working with Containers

Container Isolation

Containers from the same image are isolated from each other by default.

Process Isolation

  • Each container has its own PID namespace
  • Processes cannot see each other
  • Separate process trees

Filesystem Isolation

  • Each container has its own filesystem
  • Changes don't affect other containers
  • Writable layer is container-specific

Note: To share data between containers, use volumes or shared networks

Port Mapping

Map container ports to host ports for external access.

# Map host port 8080 to container port 80
docker run -d -p 8080:80 nginx

# Map multiple ports
docker run -d -p 8080:80 -p 8443:443 nginx

# Map to specific interface
docker run -d -p 127.0.0.1:8080:80 nginx

Format: -p HOST_PORT:CONTAINER_PORT

Environment Variables

# Pass environment variable at runtime
docker run -e MY_VAR=value myapp

# Pass multiple variables
docker run -e DB_HOST=localhost -e DB_PORT=5432 myapp

# Use environment file
docker run --env-file .env myapp

In Dockerfile

# Set default value
ENV MY_VAR=default_value

# Use in subsequent commands
RUN echo $MY_VAR

Docker Volumes

Volumes persist data beyond the container lifecycle.

# Create a named volume
docker volume create mydata

# Mount volume to container
docker run -v mydata:/app/data myapp

# Bind mount host directory
docker run -v $(pwd):/app myapp

# List volumes
docker volume ls

Use Cases: Database storage, configuration files, shared data between containers

Restart Policies

Policy Behavior
no Never restart (default)
always Always restart, including after daemon restart
unless-stopped Restart unless manually stopped
on-failure Restart only on non-zero exit code 
docker run --restart=always myapp

Module 5

Docker Networking

Docker Network Types

Type Description Use Case
bridge Default isolated network Single-host containers
host Share host's network Performance-critical apps
none No networking Isolated processing
overlay Multi-host networking Docker Swarm clusters

Custom Bridge Networks

# Create a custom network
docker network create mynetwork

# Run containers on the network
docker run -d --name web --network mynetwork nginx
docker run -d --name db --network mynetwork postgres

# Containers can reach each other by name
docker exec web ping db

# Inspect network
docker network inspect mynetwork

Benefit: Custom networks provide automatic DNS resolution between containers

Module 6

Docker Compose

Why Docker Compose?

Without Compose

  • Multiple docker run commands
  • Manual network creation
  • Complex volume management
  • Hard to reproduce setup

With Compose

  • Single YAML configuration
  • Automatic networking
  • Declarative volumes
  • One command: docker compose up

Docker Compose File

# docker-compose.yml
services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - redis
    environment:
      - REDIS_HOST=redis

  redis:
    image: redis:alpine

  db:
    image: postgres:15
    volumes:
      - db_data:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=secret

volumes:
  db_data:

Docker Compose Commands

# Start all services
docker compose up -d

# Build and start
docker compose up --build -d

# View running services
docker compose ps

# View logs
docker compose logs -f

# Stop services
docker compose down

# Stop and remove volumes
docker compose down -v

Module 7

CI/CD and Security

Docker CI Pipeline Stages

1. Test

Run unit tests

2. Build

Build Docker image

3. Scan

Security scanning

4. Push

Push to registry

Key Principle: Fail fast - catch vulnerabilities before they reach production

Security Scanning with Trivy

# Scan a Docker image
trivy image nginx:latest

# Scan with specific severity
trivy image --severity HIGH,CRITICAL nginx:latest

# Fail build on vulnerabilities
trivy image --exit-code 1 --severity CRITICAL myapp:v1

Other Tools

  • Docker Scout: Native Docker security scanning
  • Snyk: Dependency and container scanning
  • Hadolint: Dockerfile linting

Docker Security Best Practices

Image Security

  • Use specific version tags
  • Use minimal base images
  • Scan images regularly
  • Update base images

Runtime Security

  • Run as non-root user
  • Use read-only filesystems
  • Never store secrets in images
  • Limit container capabilities
# Run as non-root
RUN useradd -r appuser
USER appuser
